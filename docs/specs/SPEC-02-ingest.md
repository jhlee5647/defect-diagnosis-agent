## 1. 스토리 — 이 모듈이 하는 일

에이전트가 질문에 답하려면 "찾아볼 곳"이 미리 준비되어 있어야 한다. ingest는 `data/`의
사진+JSON 쌍 전체를 SPEC-01 파서로 읽어 **세 저장소에 나눠 적재하는 일괄 배치 작업**이다:

| 저장소 | 무엇을 넣나 | 누가 쓰나 |
|---|---|---|
| **D1** (SQLite) | 결함 1건 = 행 1개인 이력 표 (연도·단지·호기·블레이드·부위·결함종류·심각도·파일경로) | history_query — "성산 5호기 심각도 3 이상 이력" 같은 조회 |
| **V1** (Chroma) | 사진의 이미지 임베딩 + 라벨 메타 | visual_search — "이 사진과 비슷한 과거 사례" |
| **V2** (Chroma) | 전문가 설명문의 텍스트 임베딩 + 출처 메타 | knowledge_search — "라미네이트 노출 방치하면?" |

한 번 실행하면 끝나는 오프라인 작업이고, 질의 처리(온라인) 중에는 세 저장소를 **읽기만** 한다.
실행: `uv run python -m src.ingest --data-dir data` (개발 중엔 `--data-dir stub_data`)

## 2. 함정 — 왜 규칙이 필요한가 (장면으로)

**함정 1 — 컨닝.** 평가(SPEC-06)는 사진 50장을 시험 문제로 정해두고 시스템에게 4지선다를 풀게 한 뒤 채점한다.
그런데 그 50장이 검색 DB에 들어가 있으면 — visual_search는 "가장 비슷한 과거 사례"를 찾는 도구인데,
**어떤 사진과 가장 비슷한 사진은 그 사진 자기 자신이다.** 검색 1위로 자기 자신이 정답 라벨을 달고 돌아오고,
시스템은 그걸 베껴 만점을 받는다. 실력 측정이 아니라 컨닝. 그래서 시험용 50장은 **적재 단계에서부터**
검색 DB(V1·V2)에 넣지 말아야 한다. 원 지시서는 이 누수를 "감점이 아니라 실격"으로 규정했다.

**함정 2 — 깨진 파일.** 1,000개 JSON 중 1개에 필드가 빠져 있다고 하자. 대비가 없으면 500번째에서 배치가
죽고 처음부터 다시 돌린다. 반대로 에러를 조용히 삼키면 "왜 936장밖에 없지?"를 나중에 아무도 설명 못 한다.
→ 계속 진행하되, 끝나고 "몇 건을 무슨 이유로 건너뛰었다"를 보고서에 남긴다.

**함정 3 — 번호표 불일치.** "3번 = 페인트 손상" 같은 번호↔이름 매핑표가 JSON **파일마다 각자** 들어 있다.
단 한 파일이 "3번 = 오염"으로 적혀 있다면 그 결함은 DB에 엉뚱한 이름으로 들어가고, 검색·이력·채점이
전부 그 라벨을 믿게 된다. 이건 건너뛰기로 넘길 문제가 아니라 데이터 제작 오류 → 발견 즉시 멈추고 알린다.

## 3. 실행 흐름 — 명령 한 번에 일어나는 일

```
uv run python -m src.ingest --data-dir data     ← 명령 한 번
 1. data/ 스캔                                   → JSON 986개 발견
 2. 986개 하나씩 파싱 (SPEC-01 번역기)            → 깨진 파일은 건너뛰고 사유 기록
 3. 시험 문제지 분리                              → eval/testset.txt 없으면(첫 실행) 파싱 성공분에서
    50장을 결함 종류별로 — 정상 사진 계층 포함 —     50장을 층화 추출해 기록.
    고루 뽑아 기록                                 이미 있으면 그 목록 재사용
 4. 적재
    · D1 (이력 표, SQLite)  : 파싱 성공분 전부. 결함 1건=1행이라 행수는 1,204
                              (결함 2개인 사진은 행 2개 — 그래서 986 < 1,204)
    · V1·V2 (검색용, 벡터)  : 시험용 50장을 뺀 936장만          ← 함정 1 방어
 5. 사후 검증                                     → 시험용 파일명으로 V1·V2 조회, 0건 확인
 6. 결과 보고서(IngestReport) 출력                → 총 986 = 벡터 적재 936 + 시험 제외 50 + 스킵 0
```

위 건수(986·936·1,204·50 등)는 흐름 설명용 **예시 수치**다 — 실제 건수는 6단계의 IngestReport가 산출한다.

파싱(2단계)이 시험지 분리(3단계)보다 **먼저**다: "결함 종류별로 고루"라는 층화 선정은 라벨을
읽어야만 가능하고, 이 순서 덕에 파싱 실패 파일은 테스트셋 후보에서 자동으로 빠진다.
코드상 진입점은 `run_ingest(data_dir)` 하나이며 위 6단계 전체를 수행한다.
`select_testset(...)`은 3단계에서 내부적으로 호출되는 함수다 (시드 고정 → 몇 번을 다시 뽑아도 같은 50장).

**D1 테이블 `inspections`** (행 = 결함 1건, 정상 사진은 결함 없음(NULL) 행 1개):

| filename | year | site | unit | blade | side | defect_type | severity | file_path |
|---|---|---|---|---|---|---|---|---|
| 2025_sungsan_5_A_...056.jpg | 2025 | sungsan | 5 | A | LeadingEdge | La Exposure | 3 | data/... |
| 2025_sungsan_2_C_...005.jpg | 2025 | sungsan | 2 | C | PressureSide | NULL (정상) | NULL | data/... |

**V1 항목** = 임베딩(사진의 대표결함 크롭, 정상이면 전체 축소) + 메타(파일명·결함종류·심각도·단지·연도·설명문)
**V2 항목** = 임베딩(설명문 전문 — 청킹 없음, 설명문 1건 = 항목 1건) + 메타(파일명·결함종류·심각도, 출처="caption")

## 4. 규칙 — 각각이 막는 사고

| # | 규칙 | 왜 (없으면 나는 사고) |
|---|---|---|
| R1 | `eval/testset.txt`의 파일명은 V1·V2 적재에서 제외. 파일이 없으면 `select_testset`으로 **먼저 생성**(시드 고정, 결함 종류별 + 정상 사진 층화) 후 진행. 적재 완료 후 "테스트셋 파일명이 V1·V2에 0건"을 검증 쿼리로 단언. testset.txt는 파일명 목록뿐이므로 **커밋 대상** | 함정 1 — 누수는 실격. 생성→제외→사후검증 3중 가드. 정상 사진 없는 시험지는 결함유무 문항의 음성 표본이 없어 무의미. 커밋은 시험지 고정(재현성)용 |
| R2 | 테스트셋은 D1(이력 표)에는 **포함**한다 | 벡터 검색과 달리 이력 조회는 파일명을 알아야만 정답 라벨에 닿음 — 평가(SPEC-06)에서 파일명을 주지 않는 것으로 통제. 원 지시서 요구도 "벡터 DB에서 제외" |
| R3 | 파싱 실패(SPEC-01의 ValueError) 파일은 스킵하고 사유와 함께 목록 보고. 전체 중단 금지 | 함정 2 — 깨진 파일 1개가 배치를 죽이지도, 조용히 사라지지도 않게 |
| R4 | 파일 간 categories 매핑 불일치 발견 시 즉시 에러로 중단 | 함정 3 — 라벨 오염은 스킵으로 넘길 수 없는 전역 문제 (원 지시서: 발주사 보고 사항) |
| R5 | V1 임베딩 대상: 대표결함(SPEC-01 `primary_defect`) bbox 크롭, 정상 사진은 전체 축소본 | 8K 원본 통짜 임베딩은 결함이 화면의 0.1%라 유사도가 배경에 지배됨 |
| R6 | 재실행 시 storage/ 기존 인덱스를 지우고 전량 재구축 | 증분 갱신은 비스코프 — 어중간한 중복 적재보다 항상 깨끗한 전체 재구축 |
| R7 | 완료 시 IngestReport 요약 출력 (건수·스킵 사유·소요 시간·사용 모델명) | "적재됐다"는 말 대신 증거. 모델명 기록은 평가 재현성용 |
| R8 | JSON↔사진 페어링은 파일명(stem) 인덱스로 — 같은 폴더 가정 금지. 짝 사진 없는 라벨은 스킵+사유 | 실데이터는 라벨(`02.라벨링데이터/…`)과 원천(`01.원천데이터/…`)이 병렬 트리 (2026-07-11 확인) |

## 5. 엣지 케이스 (테스트 필수)

정상 사진의 D1 NULL 행 / 다중 결함 사진의 D1 다행 적재 / 테스트셋 파일이 V1·V2에 안 들어갔는지(R1 검증 쿼리) /
깨진 JSON 1개 섞인 배치의 스킵 보고 / testset.txt 없는 최초 실행(자동 생성) / 재실행 시 중복 없음(R6)

## 6. 여기서 하지 않는 것 (→ 담당)

저장소를 **읽는** 쪽 로직 (→ SPEC-03 도구) / 증분 갱신·신규 데이터 추가 (→ 비스코프) /
가이드라인·O&M 외부 문서 적재 (→ 미보유, 확보 시 V2에 추가하는 2차 작업) / VQA 채점 (→ SPEC-06)

## 7. 완료 기준

`uv run pytest tests/test_ingest.py` 통과 (임베딩은 페이크로 대체해 저장 로직만 검증)
+ `uv run python -m src.ingest --data-dir stub_data` 실행이 IngestReport를 출력하고 R1 검증을 통과.
