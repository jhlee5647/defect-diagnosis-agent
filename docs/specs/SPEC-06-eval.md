## 1. 스토리 — 이 모듈이 하는 일

**성격 규정 먼저:**
- **pytest(단위 테스트)가 아니다.** pytest는 "코드 조각이 약속대로 동작하나"를 개발 중에 확인하고,
  eval은 **완성된 시스템 전체가 얼마나 잘 진단하나**를 숫자로 재는 성적표다.
- **사람이 안 쓴다.** 터미널에서 한 번 실행하면 문제를 풀리고 채점해 표를 뱉는 배치 스크립트다.
  (사람이 쓰는 화면은 SPEC-05 demo)

이 채점기는 두 가지 질문에 숫자로 답한다:

**질문 ① — "유사 사례를 보여주면 진단이 정말 좋아지나?"** (이 시스템의 핵심 가설)
데이터에 내장된 4지선다(VQA)를 시험지로 쓴다. 두 조건 모두 시스템에 주는 사진은
**cropped_bbox로 잘라낸 확대 크롭(1920×1080)**이다 — 크롭 영역은 정답 라벨이 아니라
"점검자가 확대해 찍은 장면"(촬영 메타)으로 간주한다. 위치 문항의 보기가 크롭 좌표 기준이라
크롭을 줘야 문제 자체가 성립하고, 원본 8K 통짜는 SPEC-03 R5가 금지한다. 같은 시험을 두 조건으로 치른다:

```
(a) VLM 단독   : 사진 + 문제만 주고 gpt-4o가 풂          ← 기준선 (맨몸 실력)
(b) RAG 주입   : visual_search로 유사 사례를 찾아
                 "과거 비슷한 사진들은 이런 결함이었다"를    ← 우리 시스템의 방식
                 함께 주고 같은 문제를 풂
채점: 정답이 JSON에 있으므로 기계 채점 → 비교표

| 조건 | 결함유무 | 유형판별 | 위치 | 특징 | 전체 |
| (a)  |   ...%  |   ...%  | ...% | ...% | ...% |
| (b)  |   ...%  |   ...%  | ...% | ...% | ...% |
```
(b)가 (a)보다 높으면 "검색이 진단을 돕는다"가 측정으로 증명된 것 — 성공 기준 3 (SPEC-00).

**질문 ② — "질문마다 도구를 제대로 골랐나?"** (라우팅 검증)
UC 5종의 프리셋 질의를 실제로 실행하고, trace에 찍힌 도구 호출을 SPEC-04 §5의 **UC별 통과 제약**
(①필수 도구 포함 ②금지 도구 부재 ③순서 제약)과 기계 대조한다. 경로 완전 일치는 요구하지 않는다 —
LLM 비결정성으로 정당한 대안 경로가 나올 수 있기 때문. 특히 UC-2는 "history 외 도구를
하나라도 부르면 실패"라는 부정 조건까지 검사한다 — 성공 기준 2.

## 2. 함정 — 왜 규칙이 필요한가 (장면으로)

**함정 1 — 컨닝은 여기서 한 번 더 막는다.** 적재 단계(SPEC-02)가 시험지 50장을 검색 DB에서 뺐지만,
그 사이 누군가 재인덱싱을 잘못 돌렸다면? 채점 직전에 "시험지 파일명이 검색 DB에 0건"을 다시 확인하고,
아니면 채점을 시작조차 하지 않는다. 또한 (b) 조건에서 시스템에 **파일명을 주지 않는다** —
파일명을 주면 이력 표(D1)에서 정답 라벨을 조회하는 우회 컨닝이 가능하기 때문이다 (SPEC-02 R2와 짝).

**함정 2 — 채점기가 틀리면 전부 무의미.** 위치 문항은 좌표 변환이 끼어 있어서(원본↔크롭),
채점기가 변환을 잘못하면 시스템이 맞혀도 오답 처리된다. → 좌표는 SPEC-01 공유 구현만 쓰고,
채점 로직 자체를 pytest로 검증한다.

**함정 3 — 유리한 숫자만 보고하고 싶은 유혹.** (b)가 (a)보다 나쁘게 나올 수도 있다.
그래도 그대로 보고한다 — 이 평가의 성공은 "측정을 했다"이지 "이겼다"가 아니고,
나쁜 숫자는 검색 품질 개선(2차 과제)의 출발점 데이터가 된다.

**함정 4 — 비용 폭주.** 50장 × 최대 4문항 × 2조건 ≈ 400회 VLM 호출이다. 실수로 반복 실행하면
비용이 쌓인다. → 실행 전에 예상 호출 수·추정 비용을 출력하고, `--limit`로 소량 시험 실행을 지원한다.

## 3. 실행 흐름 — 명령 한 번에 일어나는 일

```
uv run python eval/run_eval.py [--limit 10]      ← --limit: 처음 N장만 (시험 가동용)
 1. 누수 사전 검증 : testset.txt의 50장이 V1·V2에 0건인지 조회 — 아니면 즉시 중단
 2. 비용 예고     : "VLM 약 400회 호출 예정 (추정 $X)" 출력
 3. VQA 채점     : 사진마다 크롭(1920×1080) 생성 → (a) 조건으로 4문항 → (b) 조건으로 같은 4문항
                   → 정답 대조 (모델 응답은 "a/b/c/d 한 글자" 형식을 강제, 파싱 불가 응답은 오답+별도 집계)
 4. 라우팅 검증   : UC 5종 질의를 diagnose()로 실행 → trace를 UC별 통과 제약(SPEC-04 §5)과 대조
 5. 결과 출력     : 조건×문항유형 비교표 + 라우팅 5건 통과/실패 + README에 붙일 마크다운 저장
```

## 4. 규칙 — 각각이 막는 사고

| # | 규칙 | 왜 (없으면 나는 사고) |
|---|---|---|
| R1 | 누수 사전 검증 실패 시 채점 시작 전 중단 | 함정 1 — 오염된 시험은 치르는 것 자체가 낭비 |
| R2 | 평가 대상에게 파일명·정답 라벨 제공 금지 — 주는 것은 크롭 사진(cropped_bbox 기준 1920×1080)과 문항뿐 | 함정 1 — D1 경유 우회 컨닝 차단. 크롭은 촬영 메타라 라벨이 아니며, 크롭이어야 위치 문항 채점이 성립 |
| R3 | (a)·(b)는 같은 크롭 사진·같은 문항·같은 순서로, 모델 버전 고정 기록. (b)의 visual_search 질의 임베딩도 같은 크롭 이미지 사용 | 조건 간 차이가 "few-shot 효과"만이 되게 (비교 공정성). 질의 크롭은 V1의 결함 크롭 임베딩과 스케일 정합 |
| R4 | 결과는 문항 유형별로 분해 보고, 불리한 숫자도 그대로 + 원인 분석 한 줄 | 함정 3 — 평균 하나로 뭉개면 "위치는 못 맞히는데 유형은 잘 맞힘" 같은 통찰이 사라짐 |
| R5 | 채점 로직(응답 파싱·정답 대조·좌표 변환)은 결정적 코드 + pytest 커버 | 함정 2 — 채점기 오류는 모든 숫자를 무효화 |
| R6 | 실행 전 예상 호출 수·비용 출력, `--limit` 지원 | 함정 4 — 비용 통제 |
| R7 | 라우팅 채점의 근거는 trace 로그뿐 (사람 판단 개입 없음). 기준은 SPEC-04 §5의 UC별 통과 제약(필수/금지/순서)이며 경로 완전 일치를 요구하지 않음 | 채점이 재현 가능해야 함 — 같은 로그면 같은 점수. 완전 일치 채점은 정당한 대안 경로를 실패 처리해 평가가 불안정해짐 |

## 5. 엣지 케이스 (테스트 필수)

모델 응답이 4지선다 형식 밖("정답은 아마도...") → 오답 처리 + 형식실패 카운트 /
정상 사진(위치·유형 문항 없음) → 해당 문항 스킵하고 분모에서 제외 /
cropped_bbox 없는 문서 → 긴 변 기준 축소본으로 폴백 제공(SPEC-03 R5와 동일 규칙) + 별도 카운트 /
visual_search가 0건 반환하는 사진의 (b) 조건 → few-shot 없이 진행하고 별도 표기

## 6. 여기서 하지 않는 것 (→ 담당)

(c) 전체 에이전트 조건의 VQA 평가 — (a)·(b) 비교로 핵심 가설은 충분, (c)는 2차 /
결함 종류별 정답률 분해 — 테스트 50장으론 클래스당 표본이 부족, 데이터 확장 시 2차 /
인용 정확성 수작업 검사(원 지시서 E4) — 2차 / 시험지 선정 (→ SPEC-02 `select_testset`)

라우팅 5건 판정을 README에 기록.
