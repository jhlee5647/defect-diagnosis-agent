## 1. 스토리 — 이 모듈이 하는 일

엔지니어가 애매한 결함 사진을 만나면 하는 일이 네 가지다: ① 과거에 비슷하게 생긴 사진을 뒤져보고,
② 이 설비의 예전 점검 기록을 찾아보고, ③ 결함 기준 문서를 확인하고, ④ 사진 자체를 눈으로 뜯어본다.
tools.py는 이 네 가지를 각각 함수 하나로 만든 것이다:

| 도구 | 엔지니어의 행동 | 어디를 보나 |
|---|---|---|
| T1 `visual_search` | "이거랑 비슷한 사진 있었지 않나?" | V1 (과거 사진 임베딩) |
| T2 `history_query` | "이 호기 작년 기록 좀 봐봐" | D1 (이력 표) |
| T3 `knowledge_search` | "라미네이트 노출 기준이 뭐였지?" | V2 (설명문 코퍼스) |
| T4 `vlm_analyze` | (사진을 확대해서 직접 관찰) | **지금 올라온 새 사진** |

중요한 구분: **T1~T3은 전부 "과거"만 본다.** 사용자가 방금 올린 사진은 어떤 저장소에도 없다.
그 사진을 실제로 보는 눈은 T4 하나다 — T1이 기억(회상)이라면 T4는 눈(관찰)이다.

도구는 **판단하지 않는다.** "비슷한 사례 5건, 유사도 0.61~0.58"까지가 도구의 일이고,
"유사도가 낮으니 믿기 어렵다"는 판단은 오케스트레이터(SPEC-04)의 일이다.

## 2. 함정 — 왜 규칙이 필요한가 (장면으로)

**함정 1 — 지어내는 도구.** "성산 9호기 이력 보여줘"에 기록이 0건인데 LLM이 그럴듯한 이력을 만들어내면,
그 허위가 최종 리포트에 "근거"로 인용된다. 도구 단계에서 "이력 없음"을 명시적으로 반환해야
오케스트레이터가 지어낼 재료 자체를 받지 못한다.

**함정 2 — 위험한 SQL.** T2는 LLM이 쓴 SQL을 실행한다. LLM이 실수로(또는 이상한 질의에 낚여서)
`DELETE`나 `DROP`을 쓰면 이력 DB가 날아간다. → SELECT 문만 실행 허용, 나머지는 거부.

**함정 3 — 죽는 루프.** 도구가 예외를 그대로 던지면 에이전트 루프 전체가 죽는다. 도구는 실패도
결과로 반환해야("error": "SQL 문법 오류...") 오케스트레이터가 "그럼 파라미터를 바꿔 다시"를 판단할 수 있다.

**함정 4 — 8K 사진 통짜 전송.** 원본(8256×5504)을 그대로 VLM에 보내면 비용이 크고 성능도 나빠진다
(결함은 화면의 극히 일부). → 확대 영역(cropped_bbox)이나 지정 크롭만 전송, 원본 통짜 금지.

## 3. 도구별 명세 — 입력 → 출력 장면

모든 도구의 반환은 JSON 직렬화 가능한 dict다 (Evidence Store에 그대로 기록되기 위해).

### T1 visual_search — "이거랑 비슷한 사진 찾아줘"

```
입력: 사진 (+ 선택: k=5, 결함종류 필터, 심각도 필터, 크롭 영역)
동작: 사진을 CLIP 임베딩 → V1에서 가장 가까운 k건
출력: [ { filename: "2024_gangwon_2_B_...031.jpg", defect_type: "La Exposure",
          severity: 3, similarity: 0.86, description: "…라미네이트가 노출되어…" }, × k건 ]
```
- 필터는 재검색 전략용이다. 예: 첫 검색이 뒤죽박죽이면 "부착물 계열 결함만으로 좁혀 재검색" (UC-4)
- 유사도 점수를 가공 없이 반환 — 낮아도 그대로 (판단은 오케스트레이터)

### T2 history_query — "이 설비 기록 보여줘"

```
입력: 자연어 조건 "sungsan 5호기 A블레이드 심각도 3 이상 이력"
동작: LLM이 SQL로 번역 → SELECT ... WHERE site='sungsan' AND unit=5 ... → D1 실행
출력: { rows: [ {year: 2024, defect_type: "Paint Damage", severity: 2, filename: "..."} ],
        sql: "SELECT ...", count: 1 }        ← 실행된 SQL도 반환 (검증 가능성)
0건이면: { rows: [], count: 0, message: "조회 조건에 해당하는 이력 없음" }
```

### T3 knowledge_search — "이 결함 기준이 뭐지?"

```
입력: 텍스트 질의 "라미네이트 노출 방치 시 진행"
동작: 질의 임베딩 → V2에서 유사 설명문 k건 (V2는 설명문 1건 = 항목 1건, 청킹 없음 — SPEC-02)
출력: [ { text: "노출된 라미네이트는 마모·외부 충격에 취약해…", 
          source: "2023_yeongkwang_1_A_...014.jpg의 설명문", similarity: 0.79 }, × k건 ]
```
- 출처 없는 문단 반환 금지 — 리포트의 "근거 인용"이 전부 여기서 나온다

### T4 vlm_analyze — "사진을 직접 봐줘" (시스템의 유일한 눈)

```
입력: 사진 1~2장 + 관찰 질문 (+ 선택: few-shot 사례 목록)
동작: 사진을 크롭·축소 → gpt-4o에 질문과 함께 전달
출력: { observation: "확대 영역에서 방향성 있는 섬유질 결이 관찰됨. 오염과 불일치…",
        confidence: "high|medium|low" }
```
- **few-shot 주입**: 유사 사례의 라벨·설명문을 프롬프트에 포함 — "과거 이 사진(페인트손상 sev2)은
  이런 특징이었다. 이와 비교해 현재 사진을 판별하라." T1의 검색 결과가 T4의 재료가 되는 연결이
  이 시스템의 핵심 가설(few-shot이 정답률을 올린다)이다. 사례 이미지까지 넣을지는 **구현 시 확정**(비용 대비 효과)
- 2장 입력 = 비교 모드: "작년 사진 대비 손상이 커졌는가" (UC-5)
- 크롭 좌표 계산은 SPEC-01 `to_crop_coords` 공유 구현만 사용

## 4. 규칙 — 각각이 막는 사고

| # | 규칙 | 왜 (없으면 나는 사고) |
|---|---|---|
| R1 | 도구는 판단·계획·타 도구 호출 금지. 사실(검색 결과·관찰)만 반환 | 도구가 판단하면 라우팅 채점과 자기검증이 무의미 (아키텍처 원칙, SPEC-00 §2) |
| R2 | 0건·빈 결과는 명시적 메시지로 반환. 결과 지어내기 금지 | 함정 1 — 허위 근거가 리포트에 인용됨 |
| R3 | T2는 SELECT 단일문만 실행. 그 외 구문·다중문 거부, 실행 SQL 반환 | 함정 2 — DB 파괴 방지 + 조회의 검증 가능성 |
| R4 | 도구는 예외를 던지지 않고 `{"error": 사유}` 반환 | 함정 3 — 루프가 죽지 않고 전략 변경 기회를 가짐 |
| R5 | T4에 원본 통짜 전송 금지 — cropped_bbox 크롭 또는 긴 변 기준 축소본만 | 함정 4 — 비용·성능 |
| R6 | 이미지를 여는 도구는 T1(임베딩용)·T4(관찰용)뿐. T2·T3은 텍스트만 | 이미지 흐름 경로를 2갈래로 고정 (SPEC-00 §2) |
| R7 | 모든 반환에 조회 파라미터 요약 포함 (k, 필터, 질의문) | 추론 로그에서 "무엇을 어떻게 찾았는지" 재구성 가능해야 함 |

## 5. 엣지 케이스 (테스트 필수)

T2: 0건 조회 / DELETE 시도 거부 / 잘못된 SQL의 error 반환 · T1: 필터 적용 결과 / 결과 부족(k 미달) ·
T3: 빈 코퍼스 질의 · T4: cropped_bbox 없는 사진(축소본 폴백) / few-shot 유무 프롬프트 차이
(LLM 경유부는 스모크 테스트, 결정적 부분은 pytest — SELECT 가드, 필터, 크롭 계산, 0건 처리)

## 6. 여기서 하지 않는 것 (→ 담당)

도구 선택·순서·재시도 판단 (→ SPEC-04 오케스트레이터) / 유사도 임계값 판단 (→ SPEC-04) /
저장소 만들기 (→ SPEC-02) / MMR 다양성·타일링 (→ 비스코프)

+ 스텁 인덱스 대상 도구별 실호출 스모크 1회씩의 출력 제시.
