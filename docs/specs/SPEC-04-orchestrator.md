## 1. 스토리 — 이 모듈이 하는 일

판독 데스크의 선임 엔지니어를 생각하자. 질문을 받으면 ① 답하려면 뭘 알아야 하는지 목록을 만들고,
② 필요한 조사만 골라 시키고, ③ 모인 근거가 충분한지 자문하고, ④ 부족하면 **방법을 바꿔** 다시 조사시키고,
⑤ 답안을 쓴 뒤 "내가 인용한 게 진짜 있는 자료인가"를 검증한다. 오케스트레이터는 이 사고 과정을
LangGraph 루프로 구현한 것이다. 실제 동작 장면(표준 진단, UC-1):

```
입력: 사진 + "이 사진 점검해줘. 파일명 2026_sungsan_5_A_LeadingEdge_101.jpg"

해석   : 진단 의도 → 출력은 6항목 리포트. 필요 정보: ⑴관찰 ⑵유사사례 ⑶설비이력 ⑷기준문서
         파일명 파싱 → 성산 5호기 A날개 앞전 (evidence에 기록)
반복 1 : vlm_analyze(초기 관찰) + visual_search(k=5)
         → "어두운 변색, 오염 또는 라미네이트 노출. 확신 낮음" + 유사사례 5건(라미네이트 3, 오염 2)
         → 충분한가? 아니오 — 후보가 2개로 갈림. 판별하려면 사진을 다시 봐야 함
반복 2 : vlm_analyze(확대 크롭 + 유사사례 few-shot, "섬유질 결이 있는지 관찰하라")
         → "방향성 있는 섬유질 결 관찰됨 → 라미네이트 노출, 심각도 3 추정"
         → 충분한가? 아니오 — ⑶이력 ⑷기준이 아직 없음 (리포트 6항목을 채울 수 없음)
반복 3 : history_query("sungsan 5호기 A 앞전 이력") + knowledge_search("라미네이트 노출 심각도3 조치")
         → 이력 2건 + 기준 문단 3개
         → 충분한가? 예 — 필요 정보 4개 전부 확보. 루프 종료
종결   : 초안 작성 → 자기검증(인용 파일명 4개가 evidence에 실존하는가? 심각도 판정이 관찰과 모순 없는가?)
         → 6항목 리포트 출력 + 전체 추론 로그
```

포인트: 반복 2에서 **같은 도구(vlm)를 파라미터를 바꿔** 다시 썼고(few-shot 주입), 이력 조회는
결함 종류가 좁혀진 **뒤에** 했다. 이 "상황에 따라 달라지는 순서"가 이 시스템이 파이프라인이 아니라
에이전트인 이유이고, 성공 기준 2(SPEC-00)의 검증 대상이다.

## 2. 함정 — 왜 규칙이 필요한가 (장면으로)

**함정 1 — 파이프라인으로 퇴화.** "성산 5호기 이력 보여줘"(사진 없음)에 시스템이 습관처럼 4개 도구를
전부 돌리면: 답은 이력 표 하나에 있는데 VLM 비용을 쓰고, 무관한 검색 결과가 답변을 오염시킨다.
모든 질문에 같은 순서로 도구가 돌면 이 시스템은 존재 이유가 없다 — 질문마다 경로가 달라야 하고,
그걸 로그로 증명해야 한다.

**함정 2 — 같은 실패의 반복.** "근거 부족 → 아까랑 똑같은 검색 → 여전히 부족 → 또 같은 검색…"
LLM 루프의 전형적 무한루프다. → 직전과 동일한 (도구, 파라미터) 호출을 차단해 **다르게 시도**를 강제하고,
그래도 안 되면 6회에서 멈춘다.

**함정 3 — 유창한 헛소리.** LLM은 근거가 없어도 그럴듯하게 쓴다. 초안이 evidence에 없는 파일명
"2023_sungsan_5_A_...077.jpg"를 인용하는 사고 — 검색은 다 잘 해놓고 마지막 작문에서 지어내면
앞의 모든 노력이 무효다. → 종결 단계에서 초안의 인용·수치를 evidence와 기계적으로 대조.

**함정 4 — 억지 단정.** 와류발생기 결함(전체 데이터 105장)처럼 참고 사례 자체가 빈약한 입력에서
시스템이 그럴듯하게 확정 판정을 내리면, 가장 위험한 종류의 오답이 된다. → 근거가 임계 미달이면
"전문가 확인 필요"로 정직하게 표기하는 것이 이 케이스의 **정답 동작**이다.

## 3. 실행 흐름 — 질의 1건의 생애

```
diagnose(사진?, 질문)                            ← 진입점 (demo와 eval이 호출)
 1. 해석    : 의도 파악 → 출력 형식 결정(리포트/간결답변), 필요 정보 목록화,
              파일명 있으면 파싱(SPEC-01). evidence 초기화
 2. 루프    : (최대 6회)
    a. 도구 선택 — "지금 모자란 정보가 뭐고, 어느 도구가 채우나" (이유를 로그에 기록)
    b. 호출·관찰 — 도구 실행, 결과를 evidence에 축적 (한 반복에 도구 2개까지 허용)
    c. 충분성 평가 — 필요 정보 목록 대비 빈칸 확인 → 충분하면 탈출, 부족하면 a로
 3. 종결    : (도구 호출 없음 — 모인 근거의 조립만)
    a. 초안 생성 — evidence에 있는 것만으로
    b. 자기검증 — 인용 파일명·수치가 evidence에 실존? 판정이 관찰과 모순? 근거 없는 문장 삭제/표기
    c. 형식 적용 — 진단이면 6항목 리포트, 조회·지식이면 간결 답변+출처
 4. 반환    : AgentResult { 답변, trace(반복별 로그), evidence }
```

**리포트 6항목** (진단 질의의 출력 형식): ① 결함 유무·종류 ② 위치(부위 서술) ③ 심각도(1~4)와 판단 근거
④ 유사 과거 사례 상위 3건(파일명 포함) ⑤ 해당 설비 이력(조회 결과) ⑥ 권고 조치(근거 문단 인용)

**Evidence Store**: 질의 1건 동안만 사는 스크래치패드. 도구 반환·관찰·파싱 결과가 쌓이고,
최종 답변은 **여기 있는 것만** 재료로 쓸 수 있다. 기록 주체는 오케스트레이터뿐(도구는 접근 불가).

프롬프트 문구, LangGraph 노드 분할, 충분성 평가의 세부 형식은 **구현 시 확정**.

## 4. 규칙 — 각각이 막는 사고

| # | 규칙 | 왜 (없으면 나는 사고) |
|---|---|---|
| R1 | 도구 선택은 질의·중간 결과에 따라 동적으로. 이미지 없는 질의에서 vlm·visual은 후보에서 제외 | 함정 1 — 파이프라인 퇴화. 단순 조회에 VLM 돌리면 낭비이자 라우팅 오류 |
| R2 | 직전과 동일한 (도구, 파라미터) 호출 금지 — 바꿔서 재시도하거나 다른 도구 | 함정 2 — 같은 실패 무한 반복 |
| R3 | 루프 최대 6회. 도달 시 그때까지 근거로 답하되 "근거 불충분 부분"을 명시 | 함정 2 — 탈출구 보장 + 못 채운 것도 정직하게 |
| R4 | 최종 답변은 evidence에 있는 정보만 사용. 자기검증에서 인용 파일명·수치를 evidence와 대조, 불일치는 삭제 또는 "근거 부족" 표기 | 함정 3 — 유창한 헛소리 차단 |
| R5 | 에스컬레이션: 핵심 판정의 시각 근거가 약하면(유사도 임계 미달 + VLM 확신 low) 단정 대신 "전문가 확인 필요" 리포트 | 함정 4 — 틀린 확신보다 정직한 불확실성 |
| R6 | 매 반복의 (선택 이유, 호출 파라미터, 관찰 요약, 충분성 판단)을 구조화 로그로 남겨 trace로 반환 | 경로가 달랐음을 증명할 유일한 수단 (성공 기준 2, 데모 표시, 라우팅 채점 전부 이 로그를 씀) |
| R7 | 종결 단계는 새 정보 수집 금지 (도구 호출 없음) | "검증하다 말고 또 검색"하면 루프 상한이 무의미해짐 |
| R8 | 출력 형식 분기: 이미지 진단 → 6항목 리포트 / 조회·지식 질문 → 간결 답변 + 출처 | 이력 4건 묻는 사람에게 리포트 템플릿을 들이밀지 않기 |

## 5. 유즈케이스 5종 — 기대 경로와 통과 제약 (라우팅 채점 기준)

"기대 경로"는 대표 예시다. SPEC-06의 기계 채점은 경로 완전 일치가 아니라 **통과 제약** —
①필수 도구 포함 ②금지 도구 부재 ③순서 제약 — 만 검사한다. LLM 비결정성으로 정당한 대안 경로
(예: UC-3을 검색 1회로 해결)가 나올 수 있기 때문이다. 다섯 UC의 실제 경로가 서로 달라야 한다는
성공 기준 2는 이 제약들로 여전히 증명된다 (UC-2와 UC-1의 필수/금지 집합이 겹칠 수 없음).

| UC | 입력 | 기대 경로 (예시) | 통과 제약 | 종결 형식 |
|---|---|---|---|---|
| UC-1 표준 진단 | 사진 + "점검해줘" | (vlm + visual) → vlm(few-shot 재관찰) → (history + knowledge) | 필수: 4종 전부. 순서: visual이 vlm few-shot 재관찰보다 먼저 (검색 결과가 재관찰의 재료) | 6항목 리포트 |
| UC-2 이력 조회 | "성산 5호기 A 심각도 3+ 이력" (사진 없음) | history 1회 | 필수: history. **금지: vlm·visual·knowledge — 하나라도 호출되면 실패** | 간결 답변+표 |
| UC-3 지식 질문 | "라미네이트 노출 방치하면? 심각도 등급 의미도" | knowledge → knowledge(질의 바꿔 재검색 — 2개 질문 중 하나가 첫 검색에 없음) | 필수: knowledge. 금지: vlm·visual·history (이미지도 설비 지시자도 없음) | 간결 답변+출처 |
| UC-4 희귀 결함 | 와류발생기 사진 + "문제 있나?" (파일명 없음) | (vlm + visual) → visual(클래스 필터 재검색) → knowledge → vlm(정의 대조) | 필수: vlm·visual·knowledge. 종결이 **에스컬레이션**이어야 함. history는 생략이 기대 동작(§6)이나 금지는 아님 | **에스컬레이션** 리포트 |
| UC-5 시계열 비교 | 사진 + "작년보다 심해졌나?" | (vlm + visual) → history(과거 사진 경로 조회) → vlm(두 장 비교 모드) | 필수: vlm·history. 순서: history가 vlm 2장 비교 호출보다 먼저 (과거 사진 경로가 비교의 입력) | 비교 리포트 |

## 6. 엣지 케이스 (테스트 필수)

사진 없이 "이 사진 점검해줘" → 이미지 필요 안내 / 도구가 연속 error 반환 → 상한 전 종료 + 불충분 명시 /
파일명 없는 사진(UC-4류) → 이력 조회 불가를 인지하고 생략 / 이력 0건 → 리포트 ⑤항에 "이력 없음" 명기

## 7. 여기서 하지 않는 것 (→ 담당)

도구 내부 동작 (→ SPEC-03) / 화면 표시·스트리밍 (→ SPEC-05) / 채점 (→ SPEC-06) /
도구 병렬 실행 최적화·멀티턴 대화 메모리 (→ 비스코프. "한 반복에 도구 2개"는 순차 실행으로 충분)

## 8. 완료 기준

결정적 부분(동일 호출 차단, evidence 기록, 인용 대조) pytest 통과
+ UC-2 스모크: history만 호출됐음을 trace로 확인 (전체 UC 5종 채점은 SPEC-06에서).
