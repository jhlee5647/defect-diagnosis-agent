# ADR-005 — 모델 배치: gpt-4o-mini(판단·SQL) / gpt-4o(눈)

**Status**: Accepted (2026-07-10)

## Context

시스템에서 이미지를 봐야 하는 지점은 vlm_analyze(T4) 하나뿐이다. 오케스트레이터의 도구 선택,
text-to-SQL, 충분성 판단, 자기검증, 리포트 작성은 입력이 전부 텍스트다. 원 지시서도
"라우팅·생성은 저가 텍스트 모델 / vlm_analyze만 멀티모달 모델"의 분리를 허용했다.
사용 가능한 API는 OpenAI뿐이다.

## Decision

- **오케스트레이터(A1) + text-to-SQL(T2)**: `gpt-4o-mini` — 호출 빈도가 가장 높은 곳(질의당 루프 최대 6회의 판단)이므로 저비용 모델
- **vlm_analyze(T4)**: `gpt-4o` — 관찰 품질이 진단 품질과 few-shot 가설 검증에 직결
- 모델명은 `src/config.py` 상수로 고정하고 평가 결과에 병기한다 (재현성)

## Consequences

- (+) 비용 구조 합리화 — 비싼 모델은 질의당 1~3회(관찰)만, 잦은 판단은 저가로
- (−) 모델 2종의 버전 관리 필요
- (−) 리스크: mini의 도구 선택·충분성 판단력이 부족해 라우팅 정확도가 낮게 나올 수 있음
  → 라우팅 검증(SPEC-06) 실패가 반복되면 A1만 상위 모델로 올리는 것을 1차 대응으로 한다
